# review-loop

Adversarially review "this proposal + this code change" and auto-iterate to a pass. Each round: dispatch reviewers → triage → dispatch fixes → re-review, until a pass token or the cap. Call `/review-loop` directly; for long runs wrap it in `/goal` (last section).

## Verdict normalization

Subagents routinely go off-spec (wrong token, guessed severity, no verdict line). **The main agent normalizes every verdict itself — never get dragged by the token a subagent wrote.**

**Severity** (every finding carries exactly one):

- **blocker** — broken on ship: wrong results / data corruption / crash, security hole, breaks a dependency contract or schema, core path unusable. Must fix to pass.
- **major** — serious defect: design flaw, missed failure mode or edge, clearly inconsistent with existing code, missing a key check. Fix by default.
- **minor** — local issue, doesn't affect correctness. Fix if cheap.
- **nit** — pure style / format. Usually skip.

**Verdict tokens**: pass = `APPROVE` (Reality Checker tends to write `CLEAR`, equivalent); reject = `CHANGES-REQUESTED` (`NEEDS WORK` is a synonym).

**Re-grade first**: subagent severities are often guesses. The main agent **re-grades** each finding against the definitions above — promote under-marked blocker/major (especially when a subagent rejects but labels its reason minor/nit), demote over-marked ones. **A demotion holds only if re-grading against a definition genuinely lands it lower**; if you can't name which definition the original grade got wrong and you're demoting just to terminate, it counts as unresolved.

**"Unresolved"**: a blocker/major is resolved only when **fixed**, or **explicitly demoted to minor/nit or ruled not-applicable on main-agent review**; a verbal "noted, skipping" with the severity unchanged is still **unresolved**.

**Normalization rule**: any unresolved blocker/major ⇒ counts as **reject** (even if the subagent wrote APPROVE); only minor/nit left ⇒ counts as **pass**. **Throughout this skill, "pass / reject" means the normalized verdict, not the subagent's trailing token.**

`APPROVE-DEGRADED` (degraded pass) and `CAPPED` (hit cap without passing) are **main-agent terminal tokens — not subagent tokens, never in a slot** (see Termination).

## The loop

### 1. Dispatch reviewers

Dispatch all three in parallel in one message. Each prompt is self-contained: this proposal + change; the truth source to check against (spec / contract / existing code — **the RC slot treats it as a to-be-proven baseline per §1b, not assuming the code already conforms**); what prior rounds fixed (find **new** ship-blockers); the **agreed scope** (§1d, when full requirement context exists; an out-of-scope fix must be tagged `[out-of-scope]` with a minimal in-scope alternative); the **severity definitions**; and a requirement to end with **exactly one line: a verdict token + a findings list (severity / location / explanation / fix)**. **RC-slot exception**: for the RC slot, the "what prior rounds fixed" item is stripped per §1b's input rule (§1b's local rule overrides the general instruction above).

- **Codex** — `subagent_type: codex:codex-rescue`, read-only. The only Codex path the loop can auto-dispatch (`/codex:adversarial-review` is a manual command, doesn't emit a verdict line, so the loop never calls it). The prompt must explicitly demand a verdict line (Codex returns prose by default).
- **Code Reviewer** — `subagent_type: "Code Reviewer"`. correctness / contract / edges / security / consistency with existing code. **Extra deliverable**: a guard/check checklist (with `file:line`, same categories and scope as the §1b mechanical enumeration: all four categories + scope = changed diff lines + directly-impacted unchanged call sites/helpers/cleanup/tests), serving as the prose anchor for §1b weak reconciliation. **For every guard/check/preflight/attribution point that consumes a "set dimension" (role set / config-key set / enum arm / entry-point set), add three columns**: `declared-coverage set` │ `actually-effective set` (the full per-entity set read from code) │ `equal?` — compressing "coverage ⊊ effective" from semantic reasoning into a mechanical comparison, where **not-equal = finding** (feeds §1b step 0); this forces the reviewer to confront facts like "`select_model_string("validation")` only probed the validation default" on the spot, instead of skipping them while reasoning.
- **Reality Checker** — `subagent_type: "Reality Checker"`, dispatched per the **§1b failure-enumeration pass**.

If a subagent type isn't registered, fall back to `general-purpose` (write your own strongly-adversarial prompt; for a CR fallback, carry the guard-checklist deliverable requirement). **Code Reviewer is both a semantic reviewer and the weak-reconciliation anchor**: if it returns `APPROVE` but fails to produce the guard checklist → record that slot as `not-run(missing guard checklist)`, not a pass; if the checklist is produced but missing the three columns above = equivalent to missing the checklist, same `not-run`. Domain / security-sensitive augmentation: see §1c.

> `Code Reviewer` / `Reality Checker` / `Security Engineer` / `Minimal Change Engineer` come from agency-agents (`github.com/msitarzewski/agency-agents`, dirs `engineering/` and `testing/`); `subagent_type` is each agent's frontmatter `name`; if missing locally, install them or fall back to `general-purpose`.

**Deterministic strong anchors** (optional to run the loop, required for a clean `APPROVE`). The main agent runs these **before** dispatching reviewers; hits become findings into §2, and their mechanical artifacts serve as the strong anchor for §1b completeness reconciliation. Three forms: ① **executable invariant check** (a change-impact rule for a bug category §1b has already written up — e.g. the capability-lifecycle invariant below); ② **off-the-shelf static analysis** (shellcheck / clippy / ruff / an existing CI contract check / AST or grep extraction of guard points — e.g. shellcheck the changed `*.sh`); ③ **new-dimension fan-out grep** — the main agent runs a real, reproducible `grep` (diff's new dimensions × all repo consumers × coverage-set comparison; a hit = "coverage ⊊ effective" or "same error, inconsistent exit codes across entry points") — e.g. grepping the new dimension repo-wide surfaces that `select_model_string("validation")` only probed the validation default = finding. **It counts as a strong anchor only if it leaves a mechanical artifact (command string + output) AND its pattern domain, cross-checked, ⊇ the CR three-column `actually-effective set`**:

- this ⊇ check is evaluated during §1b completeness reconciliation, **after the CR columns return** (not at grep-time);
- CR columns missing / empty → the grep counts as **weak**, no "empty-set ⊇ vacuous pass" upgrade;
- a grep pattern that omits a consumer the CR listed → **downgrade to weak**, doesn't unlock a clean `APPROVE`.

The RC's manually-listed same-name enumeration is only a weak prose anchor. **Honesty boundary**: a strong anchor only proves "scanned-set is clean," not complete (pattern completeness is not mechanically provable). **A clean `APPROVE` requires every in-scope category to have a strong anchor**; doesn't take a slot, **doesn't replace** §1b.

**Echo a status block every round** (a failed subagent returns empty, indistinguishable from "no problems"; an empty slot must not default to pass). Slots carry the **normalized** verdict:

```
This round: Code Reviewer=APPROVE │ Reality Checker(§1b)=CHANGES-REQUESTED │ Codex=not-run(empty)
Augment: none
Scope fence: agreed scope anchored │ out-of-scope findings: 0
Simplicity: net -12 flagged │ over-eng: 1 open      (or `Simplicity: no code this round, skip`)
```

The Codex slot carries only `APPROVE` / `CHANGES-REQUESTED` / `not-run(reason)`; `APPROVE-DEGRADED` appears only on the final verdict line, never in a slot.

### 1b. Failure-enumeration pass (Reality Checker slot)

Semantic reviewers (CR / Codex) tend to verify the happy path + already-written assertions, and systematically miss "failure paths × edges × false-green" (no forced enumeration structure + context pollution: read alongside the green tests / design narrative, "it was tested" becomes an anchor). So the RC slot runs the structure below + de-polluted input.

**De-polluted input**: give it the diff + truth source (spec / contract), but **strip evaluative framing** (no "already APPROVE'd N rounds," "tests pass," "design is mature," "this was fixed last round, no need to look). Mark the truth source as a **to-be-proven baseline** (use it to judge "contract claim vs observation," but don't assume the code conforms). **But give it one neutral machine-readable row ledger**: just `row ID + boolean "did this round's fix touch this row (and its deps)"`, **excluding** terminal-state values (`verified-safe` etc. are themselves verdicts; re-feeding them = an anchor) and any praise wording. The ledger answers only "which rows need re-verification," **not "which rows can be skipped"** (de-pollution ≠ exemption from re-verification).

0. **New-dimension lateral fan-out** (runs before the mechanical enumeration; decides which unchanged consumers the table must include): steps 1-2 are **vertical** (given a guard inside the diff → search down for failing inputs), and systematically miss the **lateral** class "the diff introduced a new dimension, but some unchanged consumer of it wasn't updated in sync" (preflight coverage ⊊ effective; same error, inconsistent exit codes across entry points). So first **mechanically extract the diff's new variable dimensions** (new config keys like `os.environ.get("...")`, new enum arms / `Literal` members, new role/capability constants, newly-changed `exception → exit-code` mappings), **grep every consumer of each dimension across the repo** (preflight / validation / attribution / probes / each CLI entry / `except` blocks), and pull **every consumer of the same dimension that wasn't updated in this diff** into the enumeration table. Criterion: `declared-coverage set` vs `actually-effective set` (full per-entity set), **not-equal = finding** (this step **doesn't need a whole-repo symbol index** — grepping the dimension strings within the diff + listing unchanged consumers across the repo is enough). **The executor determines anchor strength**: RC listing them by hand = weak prose anchor; the main agent running a real `grep` before dispatch + leaving the command/output = strong anchor (strong-anchor validity includes pattern domain ⊇ the CR `actually-effective set`, evaluated after the CR columns return; CR columns missing/empty → the grep counts weak). **Never judge the "config/entry fan-out" category as "covered" just because step 0 ran** — whether that category is in scope is set by "does the code have this kind of fan-out guard"; a guard present but no main-agent grep artifact → **still in scope, weak reconciliation → cap at `APPROVE-DEGRADED`** (prevents a category and its strong anchor being self-produced by the same prose step).
1. **Mechanical enumeration (list only, don't judge)**: list each ①guard / early-return / `err` branch / exception catch ②assert / validation / exit-code ③state transition (restart / reconnect / renewal / cleanup / dry-run) ④claimed-pass point (test assertion / cassette / recompute column / doctor check) into a **table with `file:line`** (including the unchanged consumers pulled in by step 0). **Scope = changed diff lines + directly-impacted unchanged code** (unchanged callers of a changed contract/signature/return semantics, called helpers, unchanged cleanup/finally paths, unchanged tests/CI whose assertions depend on changed behavior) — regressions hide in "unchanged but made reachable/dead by the diff" code.
2. **Per-row adversarial**: for each row, instantiate failing inputs `{empty / null / malformed / timeout / partial-write / restart-mid-op / concurrent dual-driver / returns-error-not-value / token-expired / zero-row query / -1 transient drop}`, writing "observed behavior vs contract claim." Any "reports success but the underlying thing broke or is lying" = blocker (fault-tolerant degradation ≠ lying — distinguish by the contract baseline). When the contract is **silent** (spec doesn't define behavior for that failing input) → record a finding "behavior undefined," don't pass by default.
3. **Out-of-table tail**: untestable acceptance / pseudo-acceptance / whole-design unprovable items the table can't enumerate — append at the end.

**Harness false-green sub-template** — when the diff contains any test / CI / soak / doctor / green-signal-producing file (judged by content intent, not directory: `tests/`, `*.sh`, `conftest.py`, test config, `*.bats`, `Dockerfile`, CI yaml, `doctor`), **additionally walk each of**: vacuous assert (always-true / asserting on the mock itself) / swallow exception then mark success / cassette recorded from a broken run / flush-writing an empty/garbage file / write fails yet marked flushed / check still `return 0` on query failure or zero rows. This is the hotspot for "the evidence itself lies," orthogonal to production failure injection, so list it separately.

**Completeness reconciliation** (= set-diff the §1b table against an anchor; "weak" below means the anchor is itself prose, so both sides can miss the same point) — the table can't be self-attested (prose reviewers *satisfice* — stop at the first good-enough reading); it needs an **anchor** to diff against:

- **Strong**: a category with an executable invariant check or a deterministic pre-gate (shellcheck/clippy/AST) — use its mechanical output as the anchor, a hit is certain. **Counts only if this round leaves a mechanical artifact** (tool/rule name + output), else weak.
- **Weak**: with no executable anchor, fall back to the CR's independent prose checklist — both sides are prose, can co-miss the same point, so **probabilistic, not complete**.
- **Bidirectional diff** (set-difference on `file:line`, mechanically decidable, mandatory; anchor points also carry `file:line`):

  | direction | meaning | terminal state |
  |---|---|---|
  | anchor ∖ table | omission (an anchor point absent from the table) | `unresolved` |
  | table ∖ anchor | unverified row (a `file:line` in neither the diff's changed lines nor the anchor) | `unresolved` — two causes: RC fabricated a row to mark verified-safe, **or** a legitimately-impacted unchanged line the anchor missed; **don't presume which** — settle a terminal state only after adding/confirming the anchor |

  **1:1 pin (hard rule, not advisory):** each anchor point maps to **exactly one** table row; a merged mega-row doesn't count as enumerating its members (else anchor ∖ table is always empty and the diff is theater). This pin holds only when the anchor is fine-grained — under weak reconciliation, a merged anchor+table is the declared incompleteness.
- **No anchor vs no strong anchor**: not even a CR prose checklist (CR failed to produce one → slot = not-run) = **no anchor → don't pass**; a prose anchor but no executable one = **weak → passes the gate, capped at degraded**.

**In-scope categories = every category the round's code actually has a guard point for** (the union of the §1b mechanical enumeration + the CR anchor + all strong anchors, **decoupled from "which categories have a strong anchor"** — scope is set by "does the code have this kind of guard"). A strong anchor's job is the gate, not defining scope. **A clean `APPROVE` requires a strong anchor for every in-scope category**; a category with a guard but no strong anchor **stays in scope** and caps at `APPROVE-DEGRADED (weak-reconciliation: incomplete)` — it's not excluded from scope, and scope must not be narrowed to "categories we wrote anchors for" (one narrow invariant can't lift a whole weak review to clean APPROVE). **Honesty boundary**: under weak reconciliation, scope-truth is still set by prose enumeration (if both prose passes miss a whole category, it silently drops out — a known limit); only a strong anchor decouples scope from prose. **Anti-vacuum**: if the in-scope set is empty (pure proposal / no code guard points), do **not** treat "every in-scope category has a strong anchor" as vacuously true to unlock a clean `APPROVE` — pure-proposal design correctness is prose-only, so cap at `APPROVE (… not-covered …)` / `APPROVE-DEGRADED`; §1b still runs (an empty table is naturally empty, not switched off), and any design / spec-vs-existing-system mismatch found and unresolved becomes a finding → reject.

**Executable invariant check** (one landing form of a strong anchor): for a **recurring bug category**, write one lightweight executable change-impact rule — pattern hit = flag + demand evidence. Executable mechanical anchors are the way out of the "under a weak-reconciliation stack, content-sufficiency isn't mechanically provable" fixpoint.

- **Example (capability-lifecycle)**: diff adds `requires_capabilities: [X]` → flag mechanically, demand evidence covering ① X's provider ② its acquisition path (eager / lazy, and the trigger timing) ③ where preflight checks X **relative to** that acquisition; any one missing → that row `unresolved`.
- **Grows incrementally**: the invariant library grows one rule per real recurring bug category, lifting that category from weak-prose fallback to strong deterministic hit; §1b prose enumeration becomes the weak fallback for categories with no invariant yet. **Honesty boundary**: it only covers categories already written up; a new category's first occurrence still slips (weak reconciliation + bot backstop catches it, then write an invariant for it). Don't let "we have an invariant library" become a new coverage false-green.

**Bot's role (all categories, backstop only)**: BugBot/Copilot is an independent backstop for all categories (including narrative drift / forward fragility). But **a bot hit ≠ that category is covered; bot silence / pass ≠ coverage proof**. So **bot findings are pure increments — not in the coverage proof, don't raise the tier, don't suppress residual disclosure**; the `not-covered` note for narrative-drift/forward-fragility is removable only when that category has a strong anchor — a probabilistic bot can't remove it. **Never** treat "ran the failure enumeration" or "the bot ran" as coverage to pass on.

**Coverage boundary**: this pass targets failure paths / edges / concurrency false-green, but under a weak-reconciliation stack even those are only probabilistic coverage; **narrative drift** (doc/spec says X, code does Y) and **forward fragility** (correct now, silently breaks later) are the orthogonal distributions a prose loop doesn't reliably cover (must carry a not-covered suffix — see Termination / evaluator ⑧).

**Each row gets one of four terminal states** (the gate checks the terminal state, not "has a disposition"), and rows turned into findings merge into §2:

- `verified-safe` — verified against each failing input, behavior matches the contract, **must attach evidence** (which failing inputs were instantiated + observed vs contract claim); a bare label counts as `unresolved`, main agent spot-audits. Two axes apart: enumeration-completeness (are all anchor points in the table) is checked by the reconciliation; evidence-sufficiency (did the failing-input set miss the one that blows up) is mechanically provable only under a strong anchor, probabilistic under weak — so even a full table of verified-safe caps at `APPROVE-DEGRADED` under a weak stack.
- `fixed` — turned into a finding and fixed (attach the fix pointer).
- `accepted-degraded` — explicitly demoted / ruled not-applicable on review, **must attach an auditable reason keyed to a severity definition** (= the row-level landing of "review-demote" from Verdict Normalization); if you can't name which definition the original grade got wrong = still `unresolved`.
- `unresolved` — not verified / not fixed / reason doesn't hold / terminal state without evidence.

**Only all rows ∈ {verified-safe, fixed} satisfies the clean `APPROVE` gate**; any `accepted-degraded` → at most `APPROVE-DEGRADED`; any `unresolved` → reject.

**Expectation setting**: for projects without a strong anchor on every in-scope category (most small repos), the **normal steady state is `APPROVE-DEGRADED (weak-reconciliation: incomplete)`** — by design, not a fault (weak reconciliation never unlocks a clean APPROVE, same tier as a structural Codex absence — see Pass tier); to get a clean APPROVE, add a strong anchor for every in-scope category.

### 1c. Adaptive augmentation (conditional / optional / findings-only / no slot)

Beyond the three slots, the main agent may **add** reviewers by change domain — but only along two **free** distribution axes, and **anti-clone** (a fourth reviewer on the same distribution ≈ 0 value, forbidden):

- **Knowledge axis (domain expert)**: **named** domain knowledge CR/RC/Codex don't have (reentrancy/checks-effects-interactions, WCAG contrast, query-plan pathology, RTOS timing, tax law…).
- **Method axis (lens)**: a Claude reviewer with a **clearly different method** from those already running (structural design / threat modeling / adversarial refutation / forward fragility), **not** yet another generic semantic reviewer.

**Deterministic trigger pre-pass** (run before dispatch, leave an auditable matrix): mechanically scan `changed-file globs × content greps × proposal subject` → matched domain → named gap/lens → registered expert? → dispatch / skip + reason. Mapping examples (not authoritative / not exhaustive, the registry wins): `*.sol`·`delegatecall`→Blockchain Security Auditor; DB migration→Database Optimizer; ARIA→Accessibility Auditor; CI/IaC→DevOps Automator; proposal/design text→Software Architect.

**Three add gates (all must hold)**: ① you can name which **new distribution** it occupies (named — "frontend-related" doesn't count); ② that domain is **actually touched by this change** (matrix hit, not "maybe"); ③ it maps to a **registered** agent, else **SKIP** — **don't fall back to `general-purpose`** (a generic clone is exactly the ≈0 fourth reviewer). **Burden of proof on the adder**: adding one you can't name a new distribution for = a defect (over-augmentation = a vacuous-assert analog); a matrix hit you skip must be justified as major-not-fixed.

**Lands as findings-only**: all augmentation **produces findings into §2 only**, **takes no slot, holds no verdict, never erases the CR guard-checklist anchor**; even a correctness-class one (e.g. `Security Engineer`) is findings-only, to preserve the three-slot invariant. **Echo** on a separate line: `Augment: +Blockchain Security Auditor(knowledge:reentrancy) … │ migrations→Database Optimizer(expert unregistered, generic coverage only)`; if none, `Augment: none`; a matrix hit whose expert is unregistered must be disclosed as `(<domain>: expert unregistered, generic coverage only)`, **carried out as a suffix on the final token** (literally checked by evaluator ⑨).

**Method-axis backfill when Codex is structurally absent**: Codex is this environment's **only** verified cross-model-family distribution; absent = that axis is unavailable (no free substitute). You may dispatch a method-axis Claude (e.g. Software Architect, to backfill Codex's structural/design view) to **reduce miss rate**, but **same-family is not cross-family → it never lifts** `APPROVE-DEGRADED (Codex structural: …)`; backfill findings are increment-only, take no third-distribution slot.

### 1d. Scope fence (when full requirement context exists, mandatory)

Prevents reviewers from raising fixes in the name of "robustness / defensiveness," triage fixing them by default, and the loop finishing (or even the coding finishing) before anyone notices it added features / config / subsystems the user never asked for (scope creep mistaken for a bug fix).

**Raise the fence (once at start)**: if the context contains **a complete requirement description the AI and user settled on** (explicit goal / acceptance / boundaries, not one vague instruction), the main agent first **extracts and restates** an "**agreed scope**" list (what's in / what's out / the interfaces and behavioral contracts the user already approved), as a scope truth source alongside code/spec. With no full context, skip (echo `Scope fence: no full requirement context, skip`) — **don't block, don't invent scope**.

**Scope binary** (triage judges each blocker/major once):

- **In-scope**: fixing a failure path that, under the agreed behavior, would cause wrong results / crash / data corruption / contract mismatch, **without adding user-visible behavior / interfaces / config / dependencies and without changing an approved contract** → proceeds to §2, fixed by default.
- **Out-of-scope (big change)**: the fix needs to **add features / config / a CLI entry / a dependency not in the requirement**, **change an approved behavior or interface contract**, or **materially expand implementation scope** (a new cross-module subsystem, a new abstraction layer, retry / persistence / concurrency / cache machinery the user didn't ask for) → **not fixed automatically**.

**Handling an out-of-scope finding (don't auto-fix + ask immediately + don't silently launder)**: triage hits one → the main agent **pauses immediately and reports to the user** (what the finding is, why fixing it crosses the agreed scope, the minimal in-scope fix vs the crossing fix, and a request for authorization).

- Before authorization, **don't dispatch a fix for it**, and **don't** silently demote / launder it as `accepted-degraded` to keep the round clean (that's another false-green).
- Authorized → update the agreed scope, turn it in-scope, fix normally; declined / narrowed → `accepted-degraded` with an auditable reason (= out of scope, user explicitly doesn't want it).

**Prompt landing (carried every round)**: the §1 reviewer prompt attaches the agreed scope and asks for out-of-scope fixes to be tagged `[out-of-scope]` with a minimal in-scope alternative; the §3 fix spec pins the scope and "only touch X, add no new Y" at the top.

**/goal interaction**: the wait-for-authorization an out-of-scope finding triggers is a **hard pause (halt-for-user)** — the round ends with `OUT-OF-SCOPE-PENDING (N left)`, **emits no pass / `CAPPED` token**, and /goal accordingly stops auto-continuing turns and hands the authorization request to the user (see Termination + the evaluator's "short-circuit first"), until the user replies. **Echo** (one line per round): `Scope fence: agreed scope anchored │ out-of-scope findings: 1 (paused for authorization: add a --retry mechanism?)`.

### 1e. Simplicity counter-pressure (ponytail lens — standing, findings-only, no slot)

The three slots are all **additive** (find missing guards / edges / failure handling); none perceives **bloat**, and the fixer (§3) is only *locally* minimal (smallest diff per finding) — it never refuses a finding, never judges a fix as over-built, never deletes stock no finding points at. So each round trends code **up**: severity drains (blocker→…→nit) while LOC climbs, and Termination fires on a diff far larger than necessary — bloat is invisible to the gate because no axis measures it. This lens is the **subtractive** distribution (orthogonal to §1b's failure-paths / false-green / narrative-drift), an advisory counter-pressure against that ratchet.

**Run** every round whose diff under review contains code (the incoming change on round 1, a fix on later rounds; a genuinely code-free proposal round → skip: `Simplicity: no code this round, skip`). Dispatch it in parallel with §1's reviewers (a general-purpose subagent carrying the embedded rubric below, or the main agent invokes `/ponytail-review` on the diff) — **findings-only, takes no slot, holds no verdict** (inherits §1c's no-slot model → preserves the three-slot invariant).

**Rubric** (ponytail-review, embedded so the loop stays self-contained): one line per finding — `file:line: <tag> <what>. <replacement>.` — tags `delete:` (dead / speculative; replacement: nothing) · `stdlib:` (hand-rolled what the stdlib ships; name it) · `native:` (dep/code the platform already does; name the feature) · `yagni:` (abstraction with one impl, config nobody sets, layer with one caller) · `shrink:` (same logic, fewer lines; show the shorter form); end with `net: -N lines possible` or `Lean already. Ship.`. Intensity **full** only (ladder: stdlib > native > installed-dep > one line > minimum code); **not ultra** — a "this requirement shouldn't exist" challenge targets explicitly-requested behavior, already off-limits via the never-simplify set below, so the loop never raises it.

**Report-only (correctness always wins — no oscillation by construction):** §1e is advisory, so **every** §1e finding is **report-only — surfaced as a finding, never auto-dispatched to the §3 fixer**. The loop never auto-deletes / -shrinks / -replaces code on §1e's say-so; a human (or a later explicitly-authorized pass) actions the cuts. This is deliberate, not timid: a findings-only prose pass can't *prove* a line dead or a `shrink:` behavior-preserving (§1b carries no whole-repo symbol index — "no caller" by grep ≠ unreachable), so auto-acting on that label is exactly the mislabel → auto-delete → re-add hazard. With **no** mid-loop mutation from §1e, the add↔delete oscillation can't start — convergence needs no monotonicity, defer, or pinning argument. Anything in ponytail's never-simplify-away set (validation at trust boundaries · error handling that prevents data loss · security · accessibility · anything explicitly requested · a hardware calibration knob · the one runnable check) isn't even reported for removal. Division of labor: bloat is **prevented** upstream (§3's ponytail ladder writes each *additive* fix lazily) and **measured/advised** here (`net: -N` + the findings) — §1e doesn't itself remove existing bloat, by design.

**Severity — advisory by default** (re-grade per Verdict Normalization): an over-engineering finding is **`minor` by default** (a quality / scope-hygiene signal, not a correctness defect — so it never blocks the pass, and a refused simplification can't stall the loop), **promotable to `major` only when the bloat itself causes a correctness or contract problem**; it **never reaches `blocker`** (bloat is not broken-on-ship). Findings **ride §2** as ordinary findings (**no new gate, no new /goal predicate**) — §1e is honest best-effort subtractive pressure, **not** an enforced gate: it has no completeness anchor, so a `Simplicity: lean` is advisory, not a proof the diff is lean. `net: -N` means **the lines §1e's own findings would remove if actioned** (a property of this pass, *not* the diff's net delta — legitimate additive guard fixes can make the diff net-positive while bloat is still deletable). **Echo** every round (run §1e only on code-touching rounds; a code-free round echoes the skip string) — one of three canonical strings: `Simplicity: net -N flagged │ over-eng: K open` / `Simplicity: lean` / `Simplicity: no code this round, skip`.

### 2. Triage

Merge findings from all sources (three slots + optional strong anchors + optional §1c augmentation + bot) **into one deduped list** (don't dispatch two fixes for the same spot), re-grade per Verdict Normalization. **Run the §1d scope binary first** (when full requirement context exists): out-of-scope blocker/major **doesn't enter the default-fix flow**, pause for authorization (don't fix unauthorized, don't silently launder as `accepted-degraded`). In-scope: blocker/major fixed by default; a major you won't fix must be **review-demoted** by the main agent (re-graded to minor/nit or ruled not-applicable per a definition) with a reason recorded — not skipped verbally with the severity unchanged; minor/nit by cost-benefit. **§1e over-engineering findings merge in here too** — all **report-only** `minor` advisory (per §1e): they are **not** auto-dispatched to the fixer (you action the cuts), and one becomes a `major` (then handled on correctness terms, not simplicity) only if the bloat itself breaks correctness/contract. Tell the user what's fixed and what's skipped.

### 3. Dispatch fixes

Fixes default to a subagent (clean context, no main-thread design bias):

- **Default** `subagent_type: "Minimal Change Engineer"` (minimal diff, refuses scope creep; falls back to `general-purpose` if unregistered) — local bug, single module, mechanical change per spec.
- **Large/complex** (cross-module / service boundary, needs rethinking architecture or data flow, schema/API migration, needs a test strategy) → `subagent_type: codex:codex-rescue`.
- **OpenSpec pure-doc mechanical fix** (wording / format / filling in an already-decided design) → main agent edits directly; anything touching requirement semantics or contract judgment routes to the two tiers above.

**The main agent writes a complete fix spec** (file path, fix, acceptance per problem); **pin the §1d agreed scope and "only touch X, add no new Y" at the top of the spec**. **Pin the ponytail ladder too** (§1e): for **each triage-approved fix** (not §1e's report-only findings, which aren't dispatched), write the laziest fix that holds — stdlib > native > installed-dep > one line > minimum code — adding no new abstraction / config / dependency unless the finding's *correctness* requires it; mark a deliberate simplification with a `ponytail:` comment naming its ceiling + upgrade path. The comment is **non-contractual context**: next round §1b reads it as declared *intent*, but **still judges that line independently against the truth source and still instantiates the failing inputs** — a real gap behind a `ponytail:` ceiling is still a finding. Never simplify away validation / error-handling / security / accessibility / explicitly-requested behavior (§1e never-simplify set). The prompt sets boundaries: do the uncontested spec items first; pause and report back only when a new discovery would make the spec wrong / introduce a regression / require expanding scope, recording adjacent small issues without self-expanding; if the fixer finds an item **can't be fixed without crossing scope** (only fixable by adding features / config / interfaces / dependencies) → **report only, don't self-implement the crossing fix** (§1d out-of-scope handling).

### 4. Re-review

Back to step 1, three reviewers in parallel (Codex may recover mid-run). "Retry every round" applies only while the loop will continue (unresolved blocker/major remain); if both Claude experts pass after normalization this round, go to Termination to judge clean / degraded / continue. "Don't run an extra round just to retry Codex once more" applies only when Codex already gave a verdict or is structural; a **transient** Codex not-run still requires a retry next round.

## Termination

A Codex result that's not a parseable verdict = "not-run," in two classes:

- **Transient** (empty return / ran but gave no verdict line) → retry next round with a stricter prompt ("end with exactly one verdict line"), **don't stop early**; can't report APPROVE-DEGRADED on this basis.
- **Structural** (not installed / not logged in / quota — won't self-recover this session) → retrying is pointless.
- **Can't tell which** → treat as transient (default to the safe side: a misjudgment only costs a few rounds, never a false pass), but capped by the K-round rule below.

**Mechanical escalation** (fixes infinite retry): persistent blockers like quota exhaustion often don't self-signal. So add a counter: Codex `not-run` for **K consecutive rounds** (default **2**, configurable; an explicit "not installed / not logged in / quota" signal judges structural on round one) → **force structural**. Escalation doesn't false-pass — it just turns infinite retry into an honest degraded termination; **every structural-degraded token is written `Codex structural: <reason>`, carrying the literal word "structural"** (so evaluator ④ hits), making /goal's "structural not-run already determined" mechanically reachable.

**An out-of-scope hard pause (§1d) takes priority over all termination judgments and the round cap** (this paragraph is the sole authority for that semantics; elsewhere stays brief). When the round has an out-of-scope blocker/major not yet adjudicated by the user, end with `OUT-OF-SCOPE-PENDING (N left)` and **emit no pass / `CAPPED` token** (mid-loop hard pause, non-terminal). This holds **even if ordinary unresolved blocker/major also exist, or the round cap is already reached** — hold the pause; ordinary unresolved items are suspended along with it and re-judged in a later round after authorization. The ending line is non-terminal, so it **carries no `narrative-drift/forward-fragility: not-covered` suffix** (disclosure re-attaches to the eventual pass / CAPPED token after authorization). **It is halt-for-user**: /goal accordingly **stops auto-continuing turns and hands the authorization request to the user** (judges neither complete nor CAPPED; ①–⑩ and the cap all inapplicable — see the evaluator's "short-circuit first"), resuming after the user authorizes / narrows. Doesn't trigger when there's no full requirement context.

**Pass gate** — any pass-class termination (`APPROVE` / `CLEAR` / `APPROVE-DEGRADED`) must satisfy: ① every §1b table row has a **terminal state**; ② passed the **bidirectional completeness reconciliation** (anchor ∖ table has no omission, table ∖ anchor has no unverified row; **not even a prose anchor = don't pass**; no strong anchor but a prose anchor doesn't block the pass, only caps at degraded); ③ findings from the table cleared §2, with no unresolved blocker/major after normalization. **"All three slots wrote APPROVE" alone is not a pass** (else the exit condition is the very false-green it's meant to catch). The gate checks enumeration-completeness + row terminal state, not "zero findings"; a legitimately-demoted finding (`accepted-degraded` with an auditable reason) doesn't block the pass but lowers the tier.

**Pass tier** (take the strictest):

| Condition | Tier |
|---|---|
| all rows ∈ {verified-safe, fixed} ∧ every in-scope category has a strong anchor ∧ in-scope set non-empty (empty = pure proposal → anti-vacuum cap at degraded) ∧ Codex gave a verdict | clean `APPROVE` |
| any `accepted-degraded` ∨ any category weak-only ∨ Codex structurally absent | at most `APPROVE-DEGRADED` (parenthesize all degrade reasons, e.g. `(weak-reconciliation: incomplete; Codex structural: not-logged-in)`) |
| any `unresolved` ∨ reconciliation fails | reject / `CAPPED` (except out-of-scope-pending, which per the Termination hard pause is held, not CAPPED) |

**Judgment**:

- **Clean termination**: all three ran (Codex gave a parseable verdict), all pass after normalization, gate satisfied, and the clean-APPROVE tier reached (all rows verified-safe/fixed + every in-scope category has a strong anchor + in-scope set non-empty) → `APPROVE` / `CLEAR`. At least two Claude experts returned a verdict; an expert (non-Codex) that didn't return is treated as incomplete (retry, or fall back to `general-purpose`) — don't terminate on a single expert's pass. Any reject → continue.
- **Codex transient not-run**: both experts pass, gate satisfied, but Codex is transient → **don't terminate, don't degrade**, retry next round until it gives a verdict / turns structural / hits the cap.
- **Degraded termination**: both experts pass, gate satisfied, but only the DEGRADED tier reached (any one of these unremovable degrade reasons appears) → **stop immediately** (re-reviewing produces no new information), `APPROVE-DEGRADED` parenthesizing all degrade reasons: **(a)** Codex structural not-run (includes the K-round mechanical escalation; §1c method-axis backfill doesn't change this tier); **(b)** any in-scope category weak-only (removable only by adding a strong anchor); **(c)** a row is `accepted-degraded`.
- **Round cap**: a configurable safety backstop (against pathological non-convergence: fixes keep introducing new problems, two reviewers deadlock), **not a "good enough, stop" quality bar**. Default **10** (raise it if too low, 10–20 common); the real end is convergence or the /goal completion condition.
  - **Cap set, must terminate at cap** (**if the round has out-of-scope-pending, the whole round ends per the Termination hard pause — not CAPPED, not subject to the cap; the below applies only with no out-of-scope-pending**): both experts pass, gate satisfied, only Codex transient → `APPROVE-DEGRADED (…: transient, cap-reached)`; unresolved blocker/major remain, gate not satisfied, or a Claude expert slot still `CHANGES-REQUESTED` / empty / not-run → `CAPPED (cap-reached, N items left)` listing the open items (terminal token, not a pass).
  - **No cap under /goal**: `CAPPED` doesn't trigger; the loop ends only on a pass token or your interrupt.

## Pairing with /goal

`/goal` checks a completion condition with a small model at the end of each turn and auto-restarts a turn if unmet, suiting long runs; it's the one stage that independently re-reads the transcript. **The status block (three slots / Augment / Scope fence / Simplicity §1e on code-touching rounds) must be echoed every round** (Simplicity is non-verdict-bearing, like Augment/Scope — never read as a slot); over-engineering findings are ordinary §2 findings already governed by the pass gate (no separate predicate); the final verdict token is echoed only on the **terminating round** (don't write it on non-terminating rounds, or it's misread as complete).

```
/goal Run an adversarial review loop on this change per review-loop; completion condition: in the latest round both Claude experts returned a verdict, the pass gate is satisfied (every §1b row has a terminal state, passed the bidirectional completeness reconciliation (anchor∖table no omission, table∖anchor no unverified row; not even a prose anchor = don't pass), no blocker/major after normalization; a clean APPROVE additionally needs all rows verified-safe/fixed + a strong anchor for every in-scope category, and any weak-only category or accepted-degraded row caps at APPROVE-DEGRADED), the final token is one of APPROVE / CLEAR / APPROVE-DEGRADED, and Codex either gave a parseable verdict or was determined structural not-run (including the K-consecutive-round mechanical escalation; transient must keep retrying); with full requirement context, echo the scope-fence line every round; echo the Simplicity §1e line on code-touching rounds (advisory, non-verdict-bearing, no predicate); and an out-of-scope (robustness adding features/config/subsystems the user didn't ask for) blocker/major must pause for user authorization before continuing — unauthorized must not pass (§1d); round cap 10 (raise it; to "loop until pass" delete this clause and set no cap), but OUT-OF-SCOPE-PENDING (§1d) is halt-for-user — when the round ends with OUT-OF-SCOPE-PENDING (N left), /goal stops auto-continuing turns and hands it to the user (judges neither complete nor CAPPED, emits no pass token, even if other ordinary items remain or the cap is reached), resuming after adjudication; otherwise (no out-of-scope-pending) at cap without a pass, terminate and record CAPPED (cap-reached, M items left)
```

The evaluator judges incompleteness **literally**, continue if any holds (each numbered item = one independently literal-checkable continue predicate). **Short-circuit first**: if the latest round ended with `OUT-OF-SCOPE-PENDING (N left)` (§1d hard pause; the status / Augment / scope-fence echo lines come first, that collar is the ending line), /goal **stops auto-continuing turns and hands it to the user — judges neither complete nor CAPPED, ①–⑩ and the cap all inapplicable** (⑩ is used only to catch the error state "out-of-scope-pending but a pass / CAPPED token was wrongly written"), resuming after the user replies. Otherwise judge by the predicates below:

- **①** the latest round is missing the three-slot status line;
- **②** any Claude expert slot is empty / not-run;
- **③** any slot is `CHANGES-REQUESTED`;
- **④** when the Codex slot is `not-run(...)`, terminating is allowed only if the final token is `APPROVE-DEGRADED` with a parenthesized reason containing the literal `structural` (every structural reason is written `Codex structural: <reason>` to guarantee a literal hit) or `transient, cap-reached`; otherwise continue (stops a transient absence being packaged as a degraded pass early);
- **⑤** the final token's subject isn't in `APPROVE` / `CLEAR` / `APPROVE-DEGRADED` (the whitelist matches by **subject, ignoring parentheses/suffix** — a suffixed form matches by subject). **Don't conflate two suffix kinds**: a **degrade suffix** (`weak-reconciliation: incomplete` / `accepted-degraded` / `Codex structural:…`) is handled by ④/⑦; a **disclosure suffix** `narrative-drift/forward-fragility: not-covered` (residual disclosure) is handled by ⑧, and its presence/absence is used only as the "bare token" test in ⑦'s empty-table branch — **not a degrade reason**, and doesn't make a non-empty table trigger ⑦'s degrade branch;
- **⑥** the §1b table is missing / has an `unresolved` row or a row with no terminal state / didn't pass the bidirectional reconciliation (including **not even a prose anchor**) → continue (weak reconciliation with a CR prose checklist counts as "has an anchor" and doesn't trigger this, only a tier cap via ⑦);
- **⑦** the final token's subject is `APPROVE` / `CLEAR` (ignoring a disclosure suffix) yet **any in-scope category is weak-only**, **or a row is `accepted-degraded`**, **or the §1b table is empty (pure proposal) and the token is a bare `APPROVE`/`CLEAR` (no `narrative-drift/forward-fragility: not-covered` etc. disclosure suffix)** → continue (these can only be `APPROVE-DEGRADED`; if the subject is already `APPROVE-DEGRADED`, doesn't trigger);
- **⑧** narrative drift / forward fragility has **no strong-anchor coverage**, and for a pass/terminal token the verdict line lacks the `narrative-drift/forward-fragility: not-covered` note → continue (a probabilistic bot can't remove that note);
- **⑨** adaptive augmentation (§1c): the `Augment: …` echo line is missing → continue; or on the terminating round the token lacks the `(<domain>: expert unregistered, generic coverage only)` suffix while that round's Augment line listed some domain as `expert unregistered` → continue. (Whether "augmentation treats an unanchored domain as covered" is a semantic judgment, not this predicate's job — augmentation not entering the coverage proof / not unlocking a clean APPROVE already structurally blocks tier-lift.)
- **⑩** scope fence (§1d): the `Scope fence: …` echo line is missing → continue (with no full context, the `… no full requirement context, skip` echo satisfies it); or the round has an out-of-scope blocker/major not yet adjudicated (`OUT-OF-SCOPE-PENDING`) yet a pass / `CAPPED` token was already written → continue (an unauthorized crossing big change must not be packaged as a pass/terminal). (Whether "a fix crosses the agreed scope" is a semantic judgment, owned by the main agent's §1d binary; this predicate only literally checks the echo line + "out-of-scope-pending ⇒ no pass".)

**Cap takes precedence** (only when a cap is set; **out-of-scope-pending is halt-for-user per the evaluator's "short-circuit first," ahead of this rule, not subject to the cap, not turned into `CAPPED`**): ①–⑨ and ⑩'s "missing scope-fence echo line" branch make the loop continue only while not at cap; otherwise (no out-of-scope-pending) **at cap, unconditionally stop working, but "stop" ≠ "pass"** — record a pass token only if the pass gate is still verifiably satisfied; **if the §1b table is missing / a row has no terminal state / the reconciliation can't be literally verified, always record `CAPPED`, never a pass token**; unresolved items remaining → record `CAPPED (cap-reached, N items left)` listing the open items. (When narrative-drift/forward-fragility has no strong-anchor coverage, a pass/CAPPED token must still carry the `narrative-drift/forward-fragility: not-covered` suffix, not waived at cap.) With no cap, `CAPPED` doesn't trigger.

> **Honesty boundary**: this is a prose instruction with no runtime enforcement; it can only **lower, not eliminate** "false-green taken as a pass" (including a silent Codex absence, an under-listed §1b table). **This loop doesn't guarantee covering narrative drift / forward fragility** — with no strong-anchor coverage (a probabilistic bot doesn't count), a pass/terminal token **must carry** the `narrative-drift/forward-fragility: not-covered` suffix (literally checked by evaluator ⑧). The real backstop is /goal's literal check + the cross-reconciliation of §1b against the CR checklist — but that waives no step: the three-slot status line / Augment / scope-fence line, severity re-grade and normalization, the pass gate, and the termination judgment all still run item by item.
