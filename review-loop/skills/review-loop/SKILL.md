# review-loop

对「本次提案 + 本次代码变更」做对抗性 review 并自动迭代到通过。每轮：派 reviewer → triage → 派修复 → 重新 review，直到放行或撞上限。直接 `/review-loop` 调用；长跑用 `/goal` 包住（见末节）。

## 结论口径

子代理常不照规范（写错 token、级别拍脑袋、不给结论行）。**主 agent 收到结论后按本节自行归一，不被子代理写的 token 带跑。**

**严重级别**（每条问题必带其一）：

- **blocker** —— 发布即坏：错误结果 / 数据损坏 / 崩溃、安全漏洞、与依赖方契约或 schema 不兼容、核心场景不可用。必须修才放行。
- **major** —— 重要缺陷：设计缺陷、漏掉的失败模式或边界、与既有代码明显不一致、缺关键校验。默认修；确实不修须按「未解决」定义复核降级，不能级别不变只记理由。
- **minor** —— 不影响正确性的局部小问题。按成本收益修。
- **nit** —— 纯风格 / 措辞 / 格式。通常不修。

**结论 token**：放行 = `APPROVE`（Reality Checker 习惯写 `CLEAR`，等价）；打回 = `CHANGES-REQUESTED`（写成 `NEEDS WORK` 同义，按打回处理）。

**先复核级别**：子代理自填的级别常拍脑袋。主 agent 按上面定义对每条问题**重新定级**——既把被低标的 blocker/major 提上来（尤其子代理打回却把理由标成 minor/nit 时），也把被高标的压下去。**下调只在按定义重判确实落到更低档时才成立**；说不出哪条定义把原档判错、纯为终止而降级，本身算未解决。

**「未解决」的定义**：一条 blocker/major 只有在**已修复**、或**主 agent 复核后显式降级为 minor/nit 或判定不适用**时才算已解决；仅口头记个理由跳过、级别不变，仍算**未解决**。

**归一硬规则**：复核后，清单含未解决 blocker/major ⇒ 该结论按**打回**计（哪怕子代理写了 APPROVE）；只剩 minor/nit ⇒ 按**放行**计。**本 skill 全程的「放行 / 打回」都指归一后结论，不是子代理末尾 token。**

`APPROVE-DEGRADED`（降级放行）与 `CAPPED`（达上限未放行）是**主 agent 专用终止 token、非子代理 token、不入任何槽**——见终止条件。

## 循环

### 1. 派 reviewer

同一条消息并行派三方。每个 prompt 自包含：本次提案与变更、对照的真理源（spec / 契约 / 既有代码；**Reality Checker 槽按 §1b 视其为待证基准、不默认代码已符合**）、前轮已修过什么（要找**新** ship-blocker；**Reality Checker 槽例外——此项按 §1b 输入规则剥离，§1b 局部规则覆盖本句**）、**级别定义**、并要求**末尾恰好一行 `APPROVE` / `CHANGES-REQUESTED` + 问题清单（级别 / 位置 / 说明 / 改法）**。

- **Codex** —— `subagent_type: codex:codex-rescue`，只读 review。这是循环唯一能自动派的 Codex 路径（`/codex:adversarial-review` 是人工命令、且不吐结论行，循环不调它）。prompt 必须明确要它给结论行（Codex 默认只返回散文）。
- **Code Reviewer** —— `subagent_type: "Code Reviewer"`。correctness / 契约 / 边界 / 安全 / 与既有代码一致性。**额外产出**：一份带 `文件:行号` 的 guard/check 清单，**类目与范围都与 §1b 机械枚举同口径**——四类目全覆盖（①guard/早返/err 分支/异常 catch ②assert/校验/退出码 ③状态转移：重启/重连/续期/cleanup/dry-run ④声称通过点：测试断言/cassette/复算列/doctor），范围 = diff 改动行 + 被改动直接波及的未改 call site/helper/cleanup/测试，作 §1b 弱对账的 prose 锚（机械枚举、低判断，不增判断负载）。**凡 guard/校验/预检/归因消费某「集合维度」（角色集 / 配置 key 集 / 枚举支 / 入口集）的每条，额外带三列**：`声明覆盖维度集`（该点自称覆盖谁）│ `实际生效维度集`（从代码读出的 per-entity 全集）│ `两者是否相等`——把「覆盖集 ⊊ 生效集」从语义推理压成机械比对列，**不等即 finding**（喂 §1b 步骤 0 扇出对账）；这一列强制 reviewer 当场面对「`select_model_string(\"validation\")` 只探了 validation 默认」这类事实，而非推理时跳过。
- **Reality Checker** —— `subagent_type: "Reality Checker"`，按 **§1b 失败枚举 pass** 派。

子代理未注册则回退 `subagent_type: general-purpose`（自写强对抗 prompt；回退 Code Reviewer 时须带上 guard 清单产出要求）。**Code Reviewer 既是语义 reviewer、又是弱对账的锚**：它若返回 `APPROVE` 却**漏产 guard/check 清单**，主 agent 归一时该槽记 `未跑完(缺 guard 清单)`、不算放行（否则缺锚的弱对账被当完整）。**清单产了、但对消费「集合维度」的点缺 §1 所述三列（声明覆盖维度集 / 实际生效维度集 / 是否相等）= 等同缺清单**，同记 `未跑完`。**领域 / 安全敏感面的专家增援见 §1c**（含 `Security Engineer`；findings-only、不占槽、不改三槽 gate）。

> `Code Reviewer` / `Reality Checker` / `Security Engineer` / `Minimal Change Engineer` 来自 agency-agents 合集：`github.com/msitarzewski/agency-agents`（`engineering/` 与 `testing/` 目录）；`subagent_type` 用各 agent frontmatter 的 `name`。本机缺失时从该 repo 安装，或先用 `general-purpose` 回退跑。

**确定性强对账锚**（对跑 loop 可选，对干净 `APPROVE` 必需）。三种形态，主 agent 派 reviewer **前**先跑、命中作为额外 finding 进 §2 triage、**且其机械产物作为 §1b 完整性对账的强对账锚**（见 §1b）：① **可执行不变量检查**——针对 §1b 已写过的 bug 类目的 change-impact 规则（见 §1b）；② **现成静态分析**——shellcheck / clippy / ruff 开 encoding 规则 / 已有 CI 契约检查 / AST 或 grep 抽 guard 点；③ **新维度扇出 grep**——**由主 agent 在派 reviewer 前用真实可复算 `grep` 命令跑**（diff 新增维度：配置 key / 枚举支 / 角色常量 / exception→退出码映射）× 全仓消费点 × 覆盖集对比，命中即「覆盖集 ⊊ 生效集」或「跨入口同错不一致」。与 ②「grep 抽 guard 点」同源，**唯有留下机械产物（grep 命令串 + 输出）、且 pattern 域经交叉校验 ⊇ Code Reviewer 三列的『实际生效维度集』，才按强锚计**（§70；**此 ⊇ 校验在 §1b 完整性对账时、CR 三列返回后评定，非 grep-time——主 agent grep 虽在派 reviewer 前先跑播种 finding，但其强/弱定级须等 CR 列到位；CR 列缺/空时 grep 一律按弱对账计，不得以「空集 ⊇ 真空通过」升强**；grep pattern 漏写某 CR 独立列出的消费点 → 降弱对账、不解锁干净 APPROVE）；RC 在 §1b 步骤 0 内跑的同名枚举只是 prose 锚（弱）、不因同名升强。它只为「配置/入口扇出」类目提供强覆盖，**不自动判该类『已覆盖』**——该类是否在范围仍由 §77 的「代码有没有该类扇出 guard」定。**诚实边界**：强锚只证『grep 命中点干净』、非『所有消费点都在 pattern 里』——pattern 完备性同属 §79『内容充分性不可机械证』不动点，CR 交叉校验抬高门槛但两 prose 仍可同漏，故扇出强锚的干净 APPROVE 是『已扫集干净』、非完备保证。**「可选」指不跑也能跑完 loop（封顶 `APPROVE-DEGRADED`）、不是「跑不跑都一样」**——**每个在范围类目都有强对账锚**才解锁干净 `APPROVE`。便宜、栈相关，多半是「规则/不变量没写」而非架构缺；不占状态槽，**不替代** §1b。

**每轮必回显一行三槽状态**（子代理失败返回空，与「无问题」不可区分；空槽不许默认成放行）。槽位写**归一后结论**、不是子代理原 token：

```
本轮 reviewer：Code Reviewer=APPROVE │ Reality Checker(§1b)=CHANGES-REQUESTED │ Codex=未跑完(空返回)
增援：无
```

Codex 槽只写 `APPROVE` / `CHANGES-REQUESTED` / `未跑完(原因)`；`APPROVE-DEGRADED` 只出现在最终聚合结论行，不入任何槽。

### 1b. 失败枚举 pass（Reality Checker 槽）

普通 prose reviewer（Code Reviewer / Codex 等语义型）倾向验 happy path + 已写断言，对「失败路径 × 边界 × 假绿」系统性漏。根因：缺**强制枚举结构** + **上下文污染**（与设计叙事、绿测同读，「测过了」成锚点）。故 Reality Checker 槽按两阶段结构跑、且**输入去污染**：

**输入规则**——prompt 给 diff + 对照真理源（spec / 契约），但**剥掉评价性框架**：不给「已 N 轮 APPROVE」「测试已过」「方案已成熟」「这块前轮已修无须再看」等任何**评价/结论性**框架（防「它没问题 / 这块已结」成锚点）。真理源标为**待证基准而非已证**（判「契约声称 vs 观察」要用它，但不默认代码已符合它）。**但区分污染与必要状态**：仍给一份**中性机读 row 台账**——只含 `行 ID + 二值「本轮 fix 是否触及该行(及其依赖)」`，**不含**具体终态值（`verified-safe` / `accepted-degraded` 本身就是 verdict，回灌即「上轮判过 safe」锚点，等于没剥）、也不含任何好评 / 「已 OK」措辞。台账只答「哪些行被本轮 fix 动过、需重验」，**不答「哪些行上轮判安全可跳过」**（去污染 ≠ 免重验）。**成本**：大表 + 小步 fix 下单轮成本 ∝ 全表（轮数仍收敛，炸的是单轮）；有强对账则未触及行交机械锚重抽、prose 只重验触及行，无锚栈认作弱对账固有成本。

0. **新维度横向扇出（先于机械枚举跑，决定表要纳入哪些未改消费点）**：根因不是枚举范围漏，是**枚举起点错**——步骤 1-2 是**纵向**的（给定 diff 内 guard → 向下找失败输入），系统性放过「diff 引入新维度、某个消费该维度的未改点没被同步扩展」这一**横向**类（preflight 覆盖集 ⊊ 生效集；同一 error 跨入口退出码不一致——bot 稳定抓、纵向枚举稳定漏）。故先从 diff **机械提取本次引入的新可变维度**：新增配置 key（`os.environ.get("PLOTPILOT_MODEL_<X>")` 类）、新增枚举成员 / `Literal` 联合支、新增角色或能力常量、新改的 `exception → 退出码 / 错误归因` 映射。**对每个新维度全仓 grep 其所有消费点**（preflight / key 校验 / 成本归因 / 探针 / 各 CLI 入口 / `except` 块），把**消费同一维度、却没在本 diff 同步更新的未改点**逐个纳入下面的枚举表——它们就是步骤 1「未改但应改」的兄弟实例，靠纵向枚举找不到、须靠本步横向定位。判据机械：`消费点声明覆盖维度集` vs `该维度实际生效集`（per-entity 全集），**两集不等 = finding**。这一步**不需全仓符号索引**，diff 内 grep 维度字符串 + 列出全仓未改消费点即可。**执行者决定锚强弱**（§70「强对账须在本轮留下机械产物：工具/规则名 + 输出，否则按弱对账计」）：RC 在本步**手工**列消费点 = **prose 锚（弱）**，完整性取决于 RC 勤勉、可漏 grep；唯有**主 agent 在派 reviewer 前用真实可复算 `grep` 命令跑**、留命令串 + 输出，才按**强对账锚**计（§45 ③；强锚有效性含 pattern 域 ⊇ CR『实际生效维度集』交叉校验，**该校验于完整性对账时、CR 列返回后评定，CR 列缺/空 → grep 按弱、不得空集真空升强**；pattern 完备性仍属 §79 不动点、非完备保证）。**严禁因 step 0 跑过就判「配置/入口扇出」类目『已覆盖』**——该类是否在范围由 §77「代码有没有该类扇出 guard」定；有 guard 但无主 agent grep 产物 → **仍在范围、仅弱对账 → 封顶 `APPROVE-DEGRADED`**（防 §77 重言式：类目与其强锚不可由同一 prose 步骤捆绑自产）。
1. **机械枚举（只列不判）**：把每个 ①guard / 早返 / `err` 分支 / 异常 catch ②assert / 校验 / 退出码判断 ③状态转移（重启 / 重连 / 续期 / cleanup / dry-run）④「声称通过」点（测试断言 / cassette / 复算列 / doctor 检查）逐个列成**带 `文件:行号` 的表**（**含步骤 0 扇出纳入的未改消费点**）。**枚举范围 = diff 改动行 + 被改动直接波及的未改代码**：改了契约/签名/返回语义的**未改调用方**、被调用的 helper、未改的 cleanup/finally 路径、断言依赖已改行为的**未改测试/CI** 都要进表——回归常藏在「未改但被 diff 变可达/变失效」的代码里，只枚举改动行的 gate 会放过它。
2. **逐行对抗**：每行实例化失败输入 `{empty / null / malformed / timeout / partial-write / restart-mid-op / 并发双驱动 / 返错而非返值 / token 过期 / 查询零行 / -1 瞬断}`，写「观察行为 vs 契约声称」。任一「报成功但底层坏了或在撒谎」= blocker（容错降级 ≠ 撒谎，按契约基准区分）。契约对某点**沉默**（spec 未规定该失败输入的行为）时，记为 finding「行为未定义、契约缺规定」，不默认放过。
3. **表外尾巴**：表枚举不到的不可测验收 / 伪验收 / 整体方案不可证，附末尾。

**harness 假绿子模板**——diff 含**任何测试 / CI-CD / soak / doctor / 产「通过·绿」信号的文件时**（按内容意图判定、不限目录；如 `tests/`、`*.sh`、`conftest.py`、`Makefile`/`pyproject` 测试配置、`*.bats`、`Dockerfile`、CI yaml、`doctor`）**额外逐项过**：vacuous assert（恒真 / 对 mock 自身断言）/ 吞异常后标成功 / cassette 录自坏掉的 run / 空刷写废档 / 写失败仍标 flushed / 检查在查询失败或零行时仍 `return 0`。这是「证据本身会撒谎」的高发区，与生产失败注入正交、须单列。

**完整性对账**：本表不可只靠自觉（prose reviewer 易 satisfice），须有**锚**比对。锚分强弱：

- **强对账**：该类目有**可执行不变量检查**（见下）或确定性前置门（shellcheck/clippy/AST）时，用其机械产物当锚——不依赖 prose、命中即确定。**强对账须在本轮留下机械产物**（工具/规则名 + 输出）才算数，否则按弱对账计。
- **弱对账**：无可执行锚时退回 Code Reviewer 独立 prose 清单当锚——两边都是 prose、可能同样合并大行 / 同点同漏，**概率性、非完备**。

**双向对账**（两向都对 `文件:行号` 集合作差、机械可判，非 prose 装修，不可省；锚点也须带 `文件:行号`）：① **锚 ∖ 表 = 漏列**（锚有表无）→ `unresolved`；② **表 ∖ 锚 = 待核行**（表里某 `文件:行号` 在 diff 改动行和锚里都查无对应）→ `unresolved`：要么 RC 编造行充 verified-safe、要么是合法被波及未改行而锚漏列，**两者都须补锚/确认后才落终态**（偏保守、不假绿，不预设是哪种）。锚里每点须映射到表里**恰好一行**，**合并大行不计**为已枚举各点（否则锚∖表恒空、对账形同虚设）；合并行的机械卡点是 1:1 映射、仅当锚为细点时生效，弱对账下锚表同样合并属已声明的非完备。

**无锚 vs 无强锚**：连 Code Reviewer prose 清单都没有（CR 漏产 → 槽记未跑完，§1）= **无任何锚 → 表不完整、gate 不放行**；有 prose 锚但无可执行锚 = **弱对账 → 过 gate、档位封顶降级**（见下）。

**强对账按类目分**：**在范围类目 = 本轮代码实际有 guard 点的全部类目**——由 §1b 机械枚举 pass（L62 四类目低判断全扫 diff + 波及）+ Code Reviewer 锚 + 全部强锚的并集共同确定，**与「哪些类目有强锚」解耦**（范围由「代码有没有该类 guard」定，不由「有没有为它写锚」定）。**强对账锚的职责是 gate、不是定义范围**——它只回答「此在范围类目是否有强覆盖」。**干净 `APPROVE` 要求每个在范围类目都有强对账锚**；某类目有 guard 但无强锚 → 它**仍在范围**、缺强锚 → gate 失败 → 封顶 `APPROVE-DEGRADED (弱对账:非完备)`（**不是把它排出范围**）。范围**不得主观圈定、不得按「已写锚的类目」反向缩窄**（缩窄被机械枚举的全类目集接住、不靠锚∖表）。一条窄不变量不能把整轮弱 review 升成干净 APPROVE。**诚实边界**：弱对账下范围真值仍由 prose 枚举定——两 prose（RC 表 + CR 锚）同时把整个类目漏掉则该类静默出范围，这是**已知 prose 依赖限制、非保证**；唯一能把范围从 prose 解耦的是强锚（代码探针），但只对有强锚的类目解耦。**反真空（纯提案）**：在范围类目集若因纯提案 / 无代码 guard 点而为空，**不得**据『每个在范围类目都有强锚』真空成立解锁干净 `APPROVE`——纯提案设计正确性是 prose-only（§79 不动点），接进评估器 ⑧ 披露族、封顶 `APPROVE (…叙事漂移/前向脆弱: 未覆盖…)` / `APPROVE-DEGRADED`；§1b 照常在跑（空表 = 自然为空、非关掉），设计 / spec-vs-既有系统不一致一经查出且未决即 finding 进 triage 打回。

**可执行不变量检查（强对账锚的落地形态之一）**：对**反复出现的 bug 类目**写一条**轻量可执行 / 机械的 change-impact 规则**：命中模式即标记 + 要求证据。它不依赖 reviewer 勤勉、也不依赖概率性 bot——**可执行机械锚（不变量检查 + 确定性静态分析）是逃出「弱对账栈下 prose 内容充分性不可机械证」不动点的手段**。

- **范例 = capability-lifecycle 不变量**：凡 diff 新增 `requires_capabilities: [X]` → 机械标记，要求证据覆盖三者：① X 的提供方；② X 的获取路径（立即 / 惰性，及触发时机）；③ preflight 查 X 相对该获取的**时序**。三者任一缺证 → 该行 `unresolved`。（「惰性获取 vs 前置检查」这类跨文件序矛盾，prose 追未改文件易 satisfice，此不变量机械命中。）
- **增长方式**：不变量库**从真实 bug 类目增量长**——每攒到一类反复漏的，就为它写一条，使该类从「弱对账 prose 回退」升级为「强对账确定性命中」；§1b prose 枚举退为**尚无不变量覆盖之类的弱回退**。
- **诚实边界**：不变量只覆盖**已写过**的类；新类首次出现仍漏（→ 弱对账 + bot backstop 接住，下次为它补不变量）。**别让「有了不变量库」成为新的覆盖假绿**——它是增量过程、不是一步到位的保证。

**强弱对账影响放行档位**（见终止条件）：**每个在范围类目都有强对账锚**方可干净 `APPROVE`；**弱对账**因完整性不可证（两 prose 同点同漏 = 已知假绿模式），**最高只到 `APPROVE-DEGRADED (弱对账:非完备)`、永不发干净 APPROVE**——与 Codex 结构性缺席同档处理，不把已知假绿模式洗成干净放行。

**预期管理**：在范围类目未全部接强对账锚的项目（多数小仓）**正常稳态就是 `APPROVE-DEGRADED (弱对账:非完备)`**——设计意图、非惩罚或故障；想要干净 `APPROVE` 就为**每个在范围类目**接强对账锚（写不变量 / 接静态分析）。

**覆盖边界（诚实声明）**：本 pass 主打失败路径 / 边界 / 并发假绿，但**弱对账栈下连这几类也只是概率性覆盖**（跨文件序/依赖矛盾，prose 追未改文件易 satisfice）。**叙事漂移**（doc/spec 说 X 代码做 Y）与**前向脆弱**（现在对、未来静默坏）更是 prose loop 不可靠覆盖的正交分布。

**bot 的定位（全类、仅 backstop）**：BugBot/Copilot 是对**所有类**（含叙事漂移 / 前向脆弱，不限于此）的**独立 backstop**。但 **bot 抓到一次 ≠ 该类已覆盖；bot 沉默 / 通过 ≠ 覆盖证明**（不完整、不可重复，且 bot 不可得时不能让保证退化成服务可用性）。故 **bot finding 纯增量、不入覆盖证明、不升档位、不抑制残留披露**——弱对账栈的诚实 token 不因 bot 而升成干净 APPROVE；叙事漂移 / 前向脆弱的 `未覆盖` 注明（评估器 ⑧）**只有该类有确定性覆盖（强对账）时才能去掉，概率性 bot 结论去不掉**。**严禁**把「跑了失败枚举」或「bot 跑过」当成覆盖放行。

输出 = 枚举表 + 每行**终态** + 表外尾巴，结论按统一格式回到 Reality Checker 槽，转 finding 的行并入 §2 triage。**行终态四选一**，gate 查的是终态、不是「有没有 disposition」（防把硬行经主观 triage 静默洗白）：

- `verified-safe` —— 该行已逐失败输入验过、行为符合契约。**须附证据**（实例化了哪些失败输入 + 观察 vs 契约声称），裸标签按 `unresolved` 计、主 agent triage 抽审。**两轴分清**：①**枚举完整性**（锚里的点是否都进表）由完整性对账查，不核验证据内容；②**证据充分性**（失败输入集有没有漏掉会炸的那个）是另一轴，只在强对账下机械可证（可执行锚直接重判该条件），弱对账下抽审概率性、属未证残留——故弱对账栈即便满表 verified-safe 也封顶 `APPROVE-DEGRADED`。
- `fixed` —— 转 finding 且已修复（附修复指向）。
- `accepted-degraded` —— 复核后显式降级 / 判不适用，**须附按严重级别定义可审计的理由**——即 §结论口径的「复核降级」在行级的落点（同义、非第二套门槛）；理由说不出哪条定义把原档判错 = 仍算 `unresolved`。
- `unresolved` —— 未验 / 未修 / 理由不成立 / 终态无证据。

**只有全部行 ∈ {`verified-safe`, `fixed`} 才满足干净 `APPROVE` 的 gate**；含 `accepted-degraded` → 最高 `APPROVE-DEGRADED`；含 `unresolved` → 打回。

### 1c. 自适应增援（条件 / 可选 / findings-only / 不占槽）

三槽之外，主 agent 可按变更域**追加** reviewer——只走两条**免费**合法分布轴，且**反克隆**（同分布第四 reviewer ≈0、禁止）：

- **知识轴（领域专家）**：CR/RC/Codex 不具备的**具名**领域知识（reentrancy/checks-effects-interactions、WCAG 对比度、query-plan 病理、RTOS 时序、税法…）。
- **方法轴（lens）**：与 CR/RC/已在跑者**方法明显不同**的 Claude reviewer（结构设计 / 威胁建模 / 对抗反驳 / 前向脆弱），**非**又一个泛化语义 reviewer。

**确定性触发 pre-pass**（派 reviewer **前**先跑、留可审计矩阵）：机械扫 `改动文件 glob × 内容 grep × 提案 subject` → 命中域 → 具名缺口/lens → 已注册专家? → 派 / skip+理由。映射示例（**非权威 / 非穷举，以注册表为准**）：`*.sol`·`delegatecall`→Blockchain Security Auditor；DB 迁移·新索引→Database Optimizer；ARIA·交互组件→Accessibility Auditor；CI/IaC→DevOps Automator；提案/设计文本→Software Architect。

**三条追加门（全满足才加）**：① 说得出占哪条**新分布**（具名，「前端相关」不算）；② 该域**确被本次变更触及**（矩阵命中、非「可能」）；③ 映射到**已注册** agent，否则 **SKIP**——**不回退 `general-purpose`**（泛化克隆正是 ≈0 的第四 reviewer）。**举证负担在追加方**：加了说不出新分布 = 缺陷（over-augmentation = vacuous assert 类比）；命中而 skip 须按 major-not-fixed 举证；`增援：…` 回显行缺失 = 评估器 ⑨ 继续（矩阵为内部 pre-pass 产物、主 agent 自核、非 ⑨ 字面项）。

**落点 = §45 强对账锚 / bot 同模型（产 finding、不占槽）**：所有增援**只产 finding 进 §2 triage**，**不占状态槽、不持槽 verdict、永不满足也永不抹掉 CR guard 清单锚**（弱对账锚只绑 Code Reviewer 自身产出）。**不并入槽 verdict**——即便 correctness 类（如 `Security Engineer`，知识轴一例）也走 finding-only 以守三槽不变量；跨域专家并 verdict 更是类别错误。**回显**另起一行、与三槽分开：`增援：+Blockchain Security Auditor(知识:reentrancy) … │ migrations→Database Optimizer(专家未注册,仅泛化覆盖)`；无则 `增援：无`；命中而专家未注册须显式披露 `(<域>: 专家未注册, 仅泛化覆盖)`（类比 Codex 未跑完、不吞），**该未注册披露随最终放行/终止 token 带出为后缀**（同 ⑧ 残留披露，由评估器 ⑨ 字面校验其存在）。

**Codex 结构性缺席时的方法轴补位**：Codex 是本环境**唯一**已验证的跨模型族分布，缺席即该轴**不可得**（无免费替代）。可选派方法轴 Claude（如 Software Architect 回补 Codex 结构/设计视角）**降漏检**，但**同族非跨族 → 绝不抬升** `APPROVE-DEGRADED (Codex 结构性: …)`，只把降级 token 做扎实；补位 finding increment-only、不占第三分布槽。

### 2. Triage

各来源问题（三方 + 可选前置门 + 可选自适应增援(§1c) + bot）**合并去重成一份**（同一处别派两套改法），按「结论口径」**复核定级**，逐条判断：blocker/major 默认修；确实不修的 major 必须主 agent **复核降级**（按定义重判为 minor/nit 或判定不适用）再记理由——不能级别不变只口头跳过；minor/nit 按收益。向用户说明修哪些、跳哪些。无值得修的问题 → 终止条件。

### 3. 派修复

修复默认派子代理（干净上下文，不带主线设计的先入为主）。按类型选执行者：

- **默认** `subagent_type: "Minimal Change Engineer"`（最小 diff、拒绝 scope creep；未注册回退 `general-purpose`）——局部 bug、单模块、照 spec 的机械改动。
- **大型复杂**（跨模块 / 服务边界、需重推架构或数据流、schema/API 迁移、需设计测试策略）`subagent_type: codex:codex-rescue`。
- **OpenSpec 纯文档机械修复**（措辞 / 格式 / 补齐已定方案）主 agent 直接改；涉及需求语义或契约判断则按上两档分流。

**主 agent 写完整修复 spec**——每问题给文件路径、具体改法、验收方式。prompt 交代边界：先做 spec 内无争议项；只有新发现会使 spec 出错 / 引入回归 / 需扩大范围时才暂停回报，其余相邻小问题只记不擅自扩。

### 4. 重新 review

回步骤 1 三方并行重试（Codex 可能中途恢复）。「每轮重试」只在循环将继续（仍有未解决 blocker/major）时适用；若两个 Claude 专家本轮**归一后**均放行，进终止条件判干净 / 降级 / 继续（干净终止另受放行 gate 约束，见下）。「不为再试一次 Codex 多跑一轮」只针对 Codex 已给结论或属结构性的情况；Codex 属**瞬态**未跑完时，按终止条件仍需下一轮重试。

## 终止条件

预期 reviewer 恒为三方；Codex 未出可解析结论一律算「未跑完」，分两类：

- **瞬态**（空返回 / 跑了但没给结论行）—— 下一轮用更严 prompt（「末尾只输出一行 verdict」）重试，**不早停**。Codex 跑了只是没吐结论行 ≠ 它否决，不可据此报 APPROVE-DEGRADED。
- **结构性**（未装 / 未登录 / 额度，本会话不会自行恢复）—— 重试无意义。
- **判不准属哪类**按**瞬态**处理（重试；后续轮出现明确信号——如「未登录」——再转结构性）。默认走安全侧：误判只多费几轮、不假放行（**但「判不准默认瞬态」受下条 K 轮机械升级封顶——连续 K 轮未跑完即强制转结构性，不得无限瞬态重试**）。

**机械升级（修无限重试 / 死循环）**：额度耗尽这类**持续性阻塞常不自吐信号**，只靠「出现『未登录』才转结构性」会在 /goal **无上限**时无限重试。故加确定性计数器：Codex `未跑完` **连续 K 轮**（默认 **2**，可配；出现明确「未装 / 未登录 / 额度耗尽」信号则**首轮即判**结构性）→ **强制转结构性**。升级**不假放行**——只是从『无限重试』变为『诚实降级终止』（两专家归一放行 + gate 满足 → `APPROVE-DEGRADED (Codex 结构性: <原因>)`；**所有结构性降级 token 一律写成 `Codex 结构性: <因>` 形式、带字面『结构性』，供评估器 ④ 字面命中**），使 /goal 完成条件『已判定结构性未跑完』**机械可达**，不论有无轮数上限。

**放行 gate**——任何**放行类终止**（`APPROVE` / `CLEAR` / `APPROVE-DEGRADED`，含到顶降级）前必须满足：① §1b 枚举表每行有**终态**（非「有无 disposition」）；② 表已过**双向完整性对账**（锚∖表无漏列、表∖锚无待核行）——**连 prose 锚都没有 = 不完整、不放行**；无强锚但有 prose 锚不阻放行、仅封顶降级；③ 表转出的 finding 走完 §2 triage、归一后无未解决 blocker/major。仅「三方都写 APPROVE」不构成放行——否则 loop 的退出条件本身就是它要抓的那种假绿。gate 卡的是**枚举完整性 + 行终态**，不是「零 finding」；finding 经复核合法降级（`accepted-degraded` 带可审计理由）后不阻塞放行，但拉低档位。

**放行档位**（取最严者）：全行 ∈ {verified-safe, fixed}、**每个在范围类目都有强对账锚（且在范围类目集非空——空集 = 纯提案，按 §77 反真空封顶降级、不真空解锁干净 APPROVE）**、Codex 给结论 → 干净 `APPROVE`；任一为 `accepted-degraded`、或**任一在范围类目仅弱对账**、或 Codex 结构性缺席 → 最高 `APPROVE-DEGRADED`（括注全部降级因，如 `APPROVE-DEGRADED (弱对账:非完备; Codex 结构性: 未登录)`）；任一 `unresolved` 或对账不可过 → 打回 / `CAPPED`。

判定：

- **干净终止**：三方本轮都实际跑成（Codex 给了可解析结论）、**归一后均放行**、**放行 gate 满足、且达干净 APPROVE 档位**（全行 verified-safe/fixed + **每个在范围类目都有强对账锚、且在范围类目集非空**；空集 = 纯提案，按 §77 反真空封顶降级、不在此终止为干净 APPROVE）→ 以 `APPROVE` / `CLEAR` 结束。至少两个 Claude 专家都返回了结论；某专家（非 Codex）未返回按未完成处理（重试或回退 `general-purpose` 补齐），不可仅凭单个专家放行就终止。任一打回即继续。
- **Codex 瞬态未跑完**：两专家归一后放行、gate 满足，但 Codex **瞬态**未跑完 → **不终止也不降级**，下一轮按瞬态规则重试，直到它给结论（→ 干净终止）、转结构性（→ 降级终止）、或撞上限。
- **降级终止**：两专家归一后均放行、gate 满足，但**只够 APPROVE-DEGRADED 档位**——即出现下列任一不可消除的降级因：**(a)** Codex 属**结构性**未跑完（判定含 §终止条件 K 轮机械升级；§1c 方法轴补位降漏检、不改本档）；**(b)** 任一在范围类目无强对账锚、只能**弱对账**（完整性不可证）；**(c)** 有行终态为 `accepted-degraded` → **立即停止**（继续重审不产新信息），结论用 `APPROVE-DEGRADED` 并**括注全部降级因**（如 `APPROVE-DEGRADED (Codex 结构性: 未登录; 弱对账:非完备)`），不报成干净 APPROVE。其中 (b) 弱对账：要消除须接入确定性前置门（强对账），否则这是该栈的固有上限、不再多跑。
- **轮数上限**：一个**可配置安全兜底**——防病态不收敛（修复反复引入新问题、两 reviewer 长期僵持），**不是「够好了就停」的质量门槛**。默认 **10 轮**，嫌不够直接调高（10–20 常见）。真正的终结是**收敛**（拿到放行 token）或 /goal 里写的完成条件，不是这个数。
  - **设了上限、到顶必须终止**（守住「不无限循环」）：两专家已放行、gate 满足、仅 Codex **瞬态**未跑完 → `APPROVE-DEGRADED (…: 瞬态, 达上限)`；仍有未解决 blocker/major、gate 未满足、或某 Claude 专家槽仍 `CHANGES-REQUESTED` / 空 / 未跑完 → `CAPPED (达上限, 剩 N 项未决)` 并列出未决项——**终止 token、明确不是放行**。
  - **/goal 下可不设上限**：此时 `CAPPED` 不触发，循环只在拿到放行 token 或你中断时结束——用「可能长跑」换「不轻易放过」。

## 与 /goal 搭配

`/goal` 是内置命令：每 turn 结束用小模型查完成条件，未达成就自动再起 turn，适合长跑。它是唯一独立复读 transcript 的一环，也是把「别静默放过 Codex 缺席 / 别跳过 gate」从口头变可检查的着力点——所以每轮三槽状态行必须明确回显；最终结论 token 只在**终止轮**回显（非终止轮不写，免得被误判为完成）。

```
/goal 按 review-loop 流程对本次变更做对抗性 review 循环；完成条件：最近一轮两个 Claude 专家都返回了结论、放行 gate 满足（§1b 表每行有终态、过双向完整性对账（锚∖表无漏列、表∖锚无待核行；连 prose 锚都没有=不完整不放行）、归一后无 blocker/major；干净 APPROVE 另需全行 verified-safe/fixed + 每个在范围类目都有强对账锚，任一类目仅弱对账或有 accepted-degraded 行则最高 APPROVE-DEGRADED）、最终结论行为 APPROVE / CLEAR / APPROVE-DEGRADED 之一、且 Codex 已给可解析结论或已判定结构性未跑完（含连续 K 轮未跑完的机械升级，免无上限死循环；瞬态须继续重试）；轮数上限 10 轮（兜底，可调高；要「跑到放行为止」就删掉本句不设上限），到顶仍未放行则终止、最终 token 记 CAPPED (达上限, 剩 M 项未决)
```

评估器按**字面**判未完成，满足任一即继续跑（每条编号 = 一个独立可字面判的 continue 谓词）：

- **①** 最近一轮缺三槽状态行；
- **②** 任一 Claude 专家槽为空 / 未跑完；
- **③** 任一槽为 `CHANGES-REQUESTED`；
- **④** Codex 槽为 `未跑完(...)` 时，仅当最终 token 为 `APPROVE-DEGRADED` 且括注原因含字面 `结构性`（所有结构性因——未登录 / 额度 / 未装——一律以 `Codex 结构性: <因>` 形式写、保证字面命中，见 §终止条件机械升级）或 `瞬态, 达上限` 才算可终止，否则一律继续（防把瞬态缺席提前包装成降级放行）；
- **⑤** 最终 token 主体不在 `APPROVE` / `CLEAR` / `APPROVE-DEGRADED` 内（白名单**按 token 主体匹配、忽略括注/后缀**——故 `APPROVE (… 未覆盖 …)`、`APPROVE-DEGRADED (Codex 结构性: …)` 等带后缀形按主体命中、不算「不在白名单」。**两类后缀别混**：**降级后缀**专指 `弱对账:非完备` / `accepted-degraded` / `Codex 结构性:…` 等降级因，由 ④/⑦ 管；**注明后缀** `叙事漂移/前向脆弱: 未覆盖` 是残留披露、由 ⑧ 管；其**有无**仅作 ⑦ 空集分支的「裸 token」判据（空表 + 无此后缀 = 裸 → 继续）、**非降级因**——不使非空表的 ⑦ 降级分支触发）；
- **⑥** §1b 枚举表缺失 / 有 `unresolved` 行或行无终态 / 未过双向完整性对账（锚∖表有漏列、或表∖锚有待核行、或**连 prose 锚都没有**）→ 继续。**注意**：弱对账有 Code Reviewer prose 清单时**算「有锚」**、不触发本条（仅档位封顶降级，由 ⑦ 管）；本条的「无锚」专指连 prose 清单都缺；
- **⑦** 最终 token 主体为 `APPROVE` / `CLEAR`（无视披露后缀）却**任一在范围类目未跑强对账**（无可执行锚、只弱对账）**或有 `accepted-degraded` 行**、**或 §1b 表为空（纯提案 / 零 guard 点）而 token 为裸 `APPROVE`/`CLEAR`（无 `叙事漂移/前向脆弱: 未覆盖` 等披露后缀）**（§77 反真空：空集不真空解锁干净 APPROVE，须带未覆盖后缀或降级——给「在范围类目集非空」一个字面评估器孪生）→ 继续（弱对账 / 降级行只能 `APPROVE-DEGRADED`；token 主体已是 `APPROVE-DEGRADED` 则不触发本条）；
- **⑧** 叙事漂移 / 前向脆弱**无确定性（强对账）覆盖**、且最终为放行/终止 token 时，结论行缺 `叙事漂移/前向脆弱: 未覆盖` 注明 → 继续（概率性 bot 结论**去不掉**该注明——bot 不入覆盖证明；只有该类有强对账锚才去）。
- **⑨** 自适应增援（§1c）：最近一轮**缺『增援：…』回显行**（§1c 要求每轮回显、无追加也须写 `增援：无`，缺 = pre-pass 没跑 / 没披露，同 ① 缺三槽行）→ 继续；或**终止轮**最终放行/终止 token **未带** `(<域>: 专家未注册, 仅泛化覆盖)` 后缀、而该轮增援回显行列了某域为 `专家未注册`（命中域无专家的残留披露须随终 token 带出，同 ⑧ 字面校验）→ 继续。**注**：『增援 prose 把无强锚域当已覆盖』属语义判断、非字面可判（§79 不动点），**不由本谓词管**——增援只产 finding、不入覆盖证明、不解锁干净 APPROVE（§1c 落点）已在结构上挡住其抬档，无需评估器再做语义核。

**上限优先**（仅当设了轮数上限时）：①–⑨ 仅在**未到上限**时使循环继续；一旦**到上限**，循环**无条件停止干活**，但「停止」≠「可分类为放行」：到顶**仅放行 gate 仍可验证满足时**才记放行 token（含 `APPROVE-DEGRADED`），**§1b 表缺失 / 行无终态 / 完整性对账不可字面核实时一律记 `CAPPED`、绝不发放行 token**（评估器停跑不等于 gate 自动通过——不许用「到顶无条件终止」把未验的表洗成放行）。仍有未决则记 `CAPPED (达上限, 剩 N 项未决)`，列出未决项。`CAPPED` 是终止 token、不算放行，评估器见它即停。（叙事漂移/前向脆弱**无确定性（强对账）覆盖**时，所记的放行/CAPPED token **仍须带** `叙事漂移/前向脆弱: 未覆盖` 后缀——记录规则、不因到顶豁免、概率性 bot 去不掉。）**不设上限**时（要「跑到放行为止」），`CAPPED` 不触发，循环按 ①–⑨ 跑到拿到放行 token 或你中断为止。

> 诚实边界：这是 prose 指令、无运行时强制，只能**降低而非消除**「假绿被当成通过」的概率（含 Codex 静默缺席、§1b 表列不全）。**且本 loop 不保证覆盖叙事漂移 / 前向脆弱**（见 §1b 覆盖边界）——这两类**无确定性（强对账）覆盖时**（概率性 bot 不算覆盖、去不掉后缀），**任何放行/终止 token 必须带注明后缀**，形如 `APPROVE (叙事漂移/前向脆弱: 未覆盖, 交外部 bot 或人工)` 或 `CAPPED (…; 叙事漂移/前向脆弱: 未覆盖)`，与 `APPROVE-DEGRADED` 的诚实括注同理；它不计入放行 gate、也不假装已查，但由评估器 ⑧ 字面校验其存在（缺则继续），使这条「须注明」从 prose 期望变为可检查项、不再是无探针的自欺。真正兜底是 /goal 的字面检查 + §1b 与 Code Reviewer 清单的交叉对账——但这不豁免任何步骤：三槽状态行、级别复核与归一、放行 gate、终止判定仍须逐条执行。
